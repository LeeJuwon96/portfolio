# 현대오토에버 AI Cloud/hCloud 공장형 플랫폼 구축·운영 면접 준비

작성 기준: 사용자가 제공한 채용공고 이미지, Notion 포트폴리오, 현재 GitHub 저장소의 문서·예시 코드·발표자료

## 1. 면접관 관점의 1차 평가

### 결론

이 포지션과의 접점은 충분하다. 가장 강한 근거는 다음 네 가지다.

1. **운영·장애 대응 경험**: 방화벽 엔지니어로 고객 문의, 원격 지원, 현장 장애 대응, 정기 점검과 운영 문서화를 수행했다.
2. **Private Cloud 학습·구축 경험**: 팀 프로젝트에서 Kolla-Ansible 기반 OpenStack AIO와 그 위의 Kubernetes 멀티 클러스터 환경을 경험했다.
3. **Linux·자동화 경험**: Cron, systemd timer, shell, Kubernetes CronJob, GitOps 기반으로 백업 작업과 설정을 관리했다.
4. **복구 중심 사고와 문서화**: Velero, etcd snapshot, PostgreSQL dump, restic, MinIO를 복구 대상별로 구분하고 Runbook을 작성했다.

반면 다음 항목은 반드시 보완해야 한다.

- OpenStack 설치·장애 처리에서 본인이 직접 수행한 범위가 아직 구체적으로 드러나지 않는다.
- Prometheus/Grafana/Alertmanager는 팀 프로젝트 환경에 있었지만 본인 담당은 백업이었다. “사용 환경을 이해한다”와 “직접 구축했다”를 구분해야 한다.
- GitLab CI/CD·Argo CD도 동일하게 팀 구현과 개인 기여를 구분해야 한다.
- mgmt MinIO의 주요 백업 bucket을 AWS S3로 offsite 복제하고 S3 Lifecycle 정책을 적용했다. 면접에서는 복제 구현과 S3에서의 실제 복구 검증 여부를 구분해 설명해야 한다.
- RPO/RTO, 백업 성공률, 복구 소요 시간 등 운영 지표가 수치화되어 있지 않다.
- AWS/OpenStack 관련 자격증은 현재 확인되지 않는다.

## 2. 채용 요건과 지원자 근거 매핑

공고상 명시된 필수역량은 “팀 내외부와 효과적으로 소통할 수 있는 역량” 하나이지만, 담당업무를 보면 Private Cloud, 운영·모니터링, 자동화·CI/CD가 사실상 기술 필수영역이다.

## 3. 60초 자기소개 예시

> 안녕하세요. 백업 파일을 만드는 데서 끝내지 않고 실제 복구 가능성까지 확인하는 클라우드 인프라 엔지니어 이주원입니다. 방화벽 엔지니어로 근무하며 고객사의 네트워크 환경을 분석하고, 장애 발생 시 원격과 현장에서 복구하며 진행 상황과 결과를 문서로 공유했습니다. 최근에는 OpenStack과 Kubernetes 기반 팀 프로젝트에서 백업·복구를 담당했습니다. Kubernetes 리소스, etcd, PostgreSQL, 파일 볼륨의 복구 특성이 서로 다르다는 점을 기준으로 Velero, etcd snapshot, pg_dump, restic을 구분하고 MinIO 저장 구조와 복구 Runbook을 정리했습니다. 현대오토에버에서도 팀과 정확하게 소통하면서, 구축된 플랫폼이 실제 장애 상황에서도 복구되고 안정적으로 운영되도록 기여하겠습니다.

주의: “OpenStack을 구축했다”라고 크게 묶어 말하기보다, 팀이 구축한 환경과 본인이 맡은 백업·복구 범위를 바로 구분한다.

## 4. 필수역량: 소통·협업 면접 질문

### Q1. 고객 또는 팀원과 기술적 의견이 달랐던 경험이 있습니까?

**평가 의도**: 경청이 성격 소개에 그치지 않고 실제 행동으로 이어지는지 본다.

**답변 포인트**:

- 상대의 요구를 먼저 다시 확인한 행동
- 기술적 사실과 제약을 어떻게 설명했는지
- 합의된 기준, 담당자, 일정이 무엇이었는지
- 결과와 후속 문서화

### Q2. 장애 상황을 비기술 담당자에게 어떻게 설명합니까?

**답변 포인트**: 원인 추정부터 말하지 말고, 영향 범위-현재 상태-임시 조치-예상 복구 시간-다음 공유 시점 순서로 설명한다.

### Q3. 본인이 해결하지 못한 문제를 언제, 어떻게 에스컬레이션합니까?

**답변 포인트**: 일정 시간 무작정 버티는 것이 아니라 영향도와 긴급도로 판단한다. 시도한 명령, 로그, 가설, 재현 조건을 정리해 도움을 요청한다.

### Q4. 팀 프로젝트에서 본인의 진행 상황을 어떻게 공유했습니까?

**근거**: 백업 대상, 저장 위치, 장애별 복구 순서, 주의사항을 Runbook과 발표자료로 정리했다.

### Q5. 본인이 작성한 문서를 다른 사람이 실제로 따라 할 수 있는지 어떻게 확인합니까?

**답변 포인트**: 사전 조건, 명령, 예상 출력, 실패 시 중단 기준, 검증, 롤백을 포함한다. 가능하면 팀원에게 문서만 보고 실행하도록 리뷰받는다.

### Q6. 팀 전체 결과와 본인의 기여를 구분해 설명해 주세요.

**모범 구조**:

- 팀 결과: OpenStack-Kubernetes-CI/CD-모니터링-보안-백업을 연결한 플랫폼
- 본인 책임: 백업 대상 분류, 백업·복구 도구 선정 및 흐름 정리, 저장소 구조, 복구 검증과 Runbook
- 협업 접점: Argo CD, Vault/Secret, Prometheus 환경과 백업 파트의 연동

### Q7. 작업 중 실수하거나 판단이 틀렸던 경험과 공유 방식은 무엇입니까?

**준비 필요**: 실제 사례 하나를 선택해야 한다. 실수를 숨기지 않은 시점, 영향 차단, 원인 확인, 재발 방지 문서 또는 체크리스트까지 말한다.

### Q8. 여러 팀이 사용하는 공장형 플랫폼에서 변경 작업을 어떻게 조율하겠습니까?

**답변 포인트**: 영향도 분석, 변경 승인, 점검 시간 합의, 백업, 롤백 기준, 사전 공지, 실행 중 상황 공유, 사후 결과 보고 순으로 답한다.

## 5. 담당업무 기반 기술 질문

### Private Cloud·OpenStack

#### Q9. OpenStack에서 사용자가 VM을 생성할 때 어떤 컴포넌트가 동작합니까?

**핵심 답변**: Keystone 인증, Glance 이미지, Nova 스케줄링·컴퓨트, Neutron 네트워크, 필요 시 Cinder 블록 스토리지의 흐름을 설명한다.

#### Q10. Private Cloud와 Public Cloud의 운영 차이는 무엇입니까?

**핵심 답변**: Private Cloud는 하드웨어, 네트워크, 용량, HA, 업그레이드와 장애 책임까지 조직이 직접 부담한다. 대신 데이터 통제와 환경 맞춤성이 높다.

#### Q11. Kolla-Ansible을 선택한 이유와 한계는 무엇입니까?

**핵심 답변**: 컨테이너화된 OpenStack 서비스를 Ansible로 반복 배포하고 설정 일관성을 얻기 좋다. 다만 네트워크·스토리지·HA 설계를 대신해 주는 도구는 아니며, 로그가 호스트와 컨테이너에 분산될 수 있어 운영 이해가 필요하다.

#### Q12. OpenStack VM이 네트워크에 연결되지 않을 때 어디부터 봅니까?

**답변 순서**: 인스턴스 상태와 console log → 포트·IP·security group → Neutron agent/OVS 또는 OVN → DHCP·L3 라우팅 → 물리 NIC/VLAN/MTU.

#### Q13. 단일 AIO OpenStack 구성의 가장 큰 한계는 무엇입니까?

**핵심 답변**: 컴퓨트와 제어·스토리지 장애 도메인이 분리되지 않아 단일 호스트 장애가 전체 서비스에 영향을 준다. 학습·검증에는 유효하지만 운영 HA 구조로 보기는 어렵다.

#### Q14. 공장형 플랫폼에서 표준 이미지를 어떻게 관리하겠습니까?

**답변 포인트**: 이미지 버전, 취약점 패치, 승인·검증 파이프라인, 롤백 가능한 이전 버전, 불변 이미지, 배포 이력을 말한다.

### Linux·시스템 운영

#### Q15. Linux 서버의 CPU 사용률이 급증하면 어떻게 진단합니까?

**답변 순서**: 장애 영향 확인 → `uptime`, `top`/`pidstat` → 프로세스·스레드 → 로그 → 최근 배포·스케줄 작업 → 임시 완화 → 원인과 재발 방지.

#### Q16. 디스크 사용률이 100%인데 큰 파일이 보이지 않으면 무엇을 확인합니까?

**핵심 답변**: inode 고갈, 삭제됐지만 프로세스가 열고 있는 파일(`lsof +L1`), 마운트 누락, 컨테이너 로그, journal 크기를 확인한다.

#### Q17. CronJob과 systemd timer를 각각 어디에 사용했으며 왜 구분했습니까?

**핵심 답변**: Kubernetes control-plane의 etcd snapshot은 CronJob, mgmt 호스트의 DB·파일·컨테이너 접근은 systemd timer가 자연스럽다. 실행 대상과 장애 도메인에 맞춰 스케줄러를 선택했다.

#### Q18. systemd 서비스가 실패하면 무엇을 확인합니까?

**핵심 답변**: `systemctl status`, `journalctl -u`, unit의 `ExecStart`, 환경 파일, 사용자·권한, 의존성, 재시작 정책과 exit code를 확인한다.

#### Q19. 운영 변경 전에 반드시 확인할 것은 무엇입니까?

**핵심 답변**: 영향 범위, 백업의 최신성과 복구 가능성, 롤백 방법, 유지보수 창, 관찰 지표, 담당자와 공유 채널.

### 모니터링·운영

#### Q20. Prometheus의 pull 방식과 Zabbix의 수집 방식 차이를 설명해 주세요.

**핵심 답변**: Prometheus는 일반적으로 exporter endpoint를 주기적으로 scrape하는 pull 모델이며 시계열·라벨 기반이다. Zabbix는 agent, SNMP, agentless 검사 등 다양한 방식을 제공하고 호스트·아이템·트리거 중심 운영에 강하다. 예외적으로 Prometheus remote_write 등 push에 가까운 경로도 있다.

#### Q21. 백업 시스템을 모니터링한다면 어떤 지표를 수집합니까?

**핵심 답변**:

- 마지막 성공 시각과 작업 성공/실패
- 소요 시간과 백업 크기
- 백업 연속 실패 횟수
- 객체 저장소 가용성·용량
- snapshot 무결성 검사 결과
- 마지막 복구 리허설 시각과 결과

#### Q22. 알람이 너무 많아 운영자가 무시하기 시작하면 어떻게 개선합니까?

**핵심 답변**: 증상 중복을 묶고, 서비스 영향 기반 severity, 지속 시간, 재알림 간격, 라우팅, 유지보수 silence, Runbook 링크를 조정한다.

#### Q23. “백업 파일 생성 성공”만으로 성공 알림을 보내면 왜 부족합니까?

**핵심 답변**: 빈 파일, 불완전 dump, 잘못된 자격증명, 손상된 snapshot일 수 있다. 크기·checksum·도구별 status와 주기적 restore까지 확인해야 한다.

#### Q24. 프로젝트의 모니터링 구성에서 본인이 직접 한 것은 무엇입니까?

**안전한 답변**: “중앙 Prometheus, Grafana, Alertmanager 구성은 팀의 모니터링 담당자가 주도했습니다. 저는 그 환경의 구조와 백업 파트 연동 지점을 이해했고, 백업 성공·실패와 마지막 성공 시각을 지표화해야 한다는 개선안을 도출했습니다.” 실제로 연동을 구현했다면 커밋·설정으로 별도 입증한다.

### DevOps·CI/CD·자동화

#### Q25. GitLab CI와 Argo CD의 역할 차이는 무엇입니까?

**핵심 답변**: GitLab CI는 테스트·빌드·스캔·이미지 push와 manifest 변경을 수행하고, Argo CD는 Git의 선언 상태를 클러스터에 지속적으로 동기화한다.

#### Q26. GitOps로 백업 스케줄을 관리한 이유는 무엇입니까?

**핵심 답변**: 스케줄과 대상도 운영 정책이므로 Git 이력, 리뷰, 동일 환경 재현, drift 확인이 필요하다. 긴급 수동 변경 후에는 Git에 반영하지 않으면 다시 덮어써질 수 있다.

#### Q27. Argo CD와 Velero는 서로 대체 관계입니까?

**핵심 답변**: 아니다. Argo CD는 Git에 선언한 desired state를 복원하지만, 런타임에 생성된 객체·Secret·상태 데이터는 Git에 없을 수 있다. Velero는 클러스터 API 리소스 상태 백업으로 이를 보완한다. 데이터 볼륨과 DB는 별도 전략이 필요하다.

#### Q28. 배포 후 장애가 발생하면 어떤 기준으로 롤백합니까?

**답변 포인트**: 사전 정의한 SLI와 오류율, latency, 핵심 기능 실패, 변경 시점 상관관계, 데이터 마이그레이션의 역호환성을 본다. 이미지 태그와 manifest를 이전 검증 버전으로 되돌리고 원인 분석을 이어간다.

#### Q29. 자동화 스크립트에서 `set -euo pipefail`은 무엇을 막습니까?

**핵심 답변**: 명령 실패 무시, 미정의 변수 사용, 파이프라인 중간 명령 실패가 숨는 문제를 줄인다. 다만 예상 가능한 non-zero 반환은 명시적으로 처리해야 한다.

#### Q30. 여러 클러스터에 같은 설정을 배포할 때 어떻게 차이를 관리합니까?

**핵심 답변**: 공통 base와 환경별 overlay, 변수·Secret의 외부화, 버전 고정, 단계적 배포와 검증, 롤백 가능한 Git 이력을 사용한다.

## 6. 우대사항 및 프로젝트 심층 질문

### Q31. 왜 백업 도구를 하나로 통일하지 않았습니까?

**모범 답변**:

> 복구 단위가 달랐기 때문입니다. Kubernetes API 리소스는 Velero, control-plane 전체 상태는 etcd snapshot, PostgreSQL 데이터는 논리 dump, 호스트 볼륨은 restic이 적합했습니다. 도구 수를 줄이는 것보다 장애 시 필요한 단위로 안전하게 복구할 수 있는지를 우선했습니다.

### Q32. Velero에서 `snapshotVolumes: false`로 설정한 이유와 영향은 무엇입니까?

**핵심 답변**: 이 구성에서 Velero는 namespace API 리소스 복구에 집중하며 PV 데이터는 포함하지 않는다. 따라서 DB와 파일은 pg_dump, tar, restic 같은 별도 백업으로 복구해야 한다.

### Q33. etcd snapshot과 Velero backup의 차이는 무엇입니까?

**핵심 답변**: Velero는 선택한 Kubernetes 리소스 단위 복구에 적합하고, etcd snapshot은 클러스터 control-plane 상태 전체와 관련된다. etcd restore는 영향 범위가 크므로 격리 환경에서 rehearsal 해야 한다.

### Q34. etcd 백업 파일이 복구 가능하다고 어떻게 검증했습니까?

**현재 사실에 맞는 답변**: snapshot 파일 생성, SHA256 checksum, `etcdutl snapshot status`, 격리 경로 restore rehearsal까지 구분해서 설명한다. 공유 운영 클러스터 전체 restore를 실제 수행했다고 말하면 안 된다.

### Q35. PostgreSQL volume 자체가 아니라 `pg_dump`를 선택한 이유는 무엇입니까?

**핵심 답변**: 논리 백업은 schema·table 선택 복구와 검증이 쉽다. 반면 대용량 DB에서는 복구 시간이 길고 모든 물리 상태를 보존하지 않으므로, 규모와 RTO에 따라 physical backup·WAL/PITR을 함께 고려해야 한다.

### Q36. `pg_dump` 중에도 쓰기가 계속되면 일관성이 보장됩니까?

**핵심 답변**: PostgreSQL의 MVCC snapshot을 기반으로 dump 시점의 일관된 뷰를 얻지만, 여러 DB나 외부 파일 저장소와의 원자적 일관성은 별도 문제다. globals, storage 파일, 애플리케이션 쓰기 정지 또는 일관된 시점 설계가 필요하다.

### Q37. restic을 선택한 이유와 복구 시 필수 조건은 무엇입니까?

**핵심 답변**: 암호화, 중복 제거, 증분 snapshot, 선택 복구가 장점이다. repository와 password가 모두 필요하고 정기 `check`, 보관 정책, 임시 위치 restore 검증이 필요하다.

### Q38. MinIO가 mgmt 서버와 함께 장애 나면 어떻게 합니까?

**핵심 답변**: 동일 장애 도메인의 MinIO만으로는 복구할 수 없다. 별도 계정·리전·장애 도메인의 object storage로 offsite 복제하고, 객체 버전·삭제 보호·암호화·복구 권한을 관리해야 한다.

### Q39. 이 구조는 3-2-1 백업 원칙을 만족합니까?

**안전한 답변**: 완전 충족이라고 단정하지 않는다. 원본과 MinIO, AWS S3 사본이 실제로 독립 저장매체·장애 도메인에 있고 정기 복구가 검증되어야 한다. 현재 프로젝트는 그 원칙에 가까운 구조를 목표로 했다고 표현한다.

### Q40. AWS S3 복제를 실제로 구현했습니까?

**사실 기반 답변**:

> mgmt MinIO에 저장된 주요 백업 bucket을 AWS S3로 복제해 offsite 사본을 확보했고, S3에는 Lifecycle 정책을 적용해 오래된 객체를 Standard-IA와 Glacier 계층으로 전환하도록 구성했습니다. 이를 통해 MinIO와 mgmt 서버가 같은 장애의 영향을 받더라도 외부 저장소에 백업본이 남도록 했습니다. 다만 AWS S3 사본을 이용한 전체 재해복구를 실제 수행했는지는 복제 구현과 별도로 구분해 설명하겠습니다.

면접에는 `minio-offsite-s3-backup.sh`, systemd timer, 실행 로그, AWS S3 객체 목록과 Lifecycle 설정 화면 중 공개 가능한 증거를 준비한다.

### Q41. 백업 시간대를 순차적으로 나눈 이유는 무엇입니까?

**핵심 답변**: CPU·I/O·네트워크 부하 중첩과 데이터 의존성을 줄이고, 장애 시 어느 작업이 실패했는지 구분하기 위해서다. 다만 프로젝트 시간대는 발표 검증 편의를 위한 것이며 실제 운영은 트래픽과 RPO를 기준으로 정해야 한다.

### Q42. 이 프로젝트의 RPO와 RTO는 얼마입니까?

**현재 답변**: 정확한 수치를 측정하지 못했다는 점을 인정한다. 일일 백업이면 최악의 RPO가 약 24시간일 수 있지만 데이터 종류와 실패 감지 시간에 따라 달라진다. RTO는 실제 복구 리허설 시간을 재야 한다. 임의 수치를 만들지 말고 개선 항목으로 제시한다.

### Q43. Secret을 Velero로 백업하는 것이 안전합니까?

**핵심 답변**: Kubernetes Secret은 기본적으로 base64 인코딩일 뿐이며 저장소 암호화와 접근통제가 중요하다. 가능하면 Vault/External Secrets를 원천으로 삼아 재생성하고, 백업 자체도 암호화·최소 권한·키 분리·감사를 적용한다.

### Q44. 복구 순서를 어떻게 정했습니까?

**핵심 답변**: 기반 인프라와 인증·네트워크 → object storage와 설정 → 관리 플랫폼 → DB·파일 데이터 → 애플리케이션 리소스 → 외부 접근 → 모니터링 순으로 의존성을 고려한다. 실제 Runbook에서는 장애 유형별로 불필요한 전체 복구를 피한다.

### Q45. 본 프로젝트에서 가장 아쉬운 점은 무엇입니까?

**모범 답변**:

> 백업 종류와 절차는 정리했지만 RPO/RTO를 수치로 정의하지 못했고, 정기 복구 리허설과 백업 실패 알림을 완전 자동화하지 못했습니다. 다음 단계에서는 마지막 성공 시각, 소요 시간, 크기, 무결성 결과를 Prometheus 지표로 노출하고 Alertmanager와 Runbook을 연결하겠습니다.

### Q46. 자동차 공장 플랫폼에 적용할 때 무엇을 더 고려하겠습니까?

**답변 포인트**: 생산 중단 영향, OT/IT 망 분리, 변경 승인, 유지보수 창, 공장별 표준화와 예외, 현장 원격 접근 통제, 공급망 보안, 장기 지원 버전, 지역·공장별 DR, 복구 순서와 책임자를 말한다.

## 8. 면접 전 포트폴리오에서 반드시 고칠 사항

### 최우선

1. **S3 구현 증거 보강**: MinIO→AWS S3 복제 스크립트·timer·실행 로그·S3 객체와 Lifecycle 설정 중 공개 가능한 증거를 연결한다.
2. **팀 기여와 개인 기여 구분**: OpenStack, CI/CD, 모니터링은 팀 결과와 본인 행동을 별도 표기한다.
3. **민감정보·내부정보 재점검**: 공개 자료에 사설 IP, 사용자명, 로컬 개인키 경로, 실제 서버 경로가 포함되어 있다. `resource_runtime_report_aio_mgmt (1).html`, 발표 PDF, `backup-presentation.md`를 우선 확인한다.
4. **검증 증거 추가**: 실행 명령, 예상 출력, 실패 사례, 실제 복구 전후 검증을 민감정보 제거 후 캡처한다.

### 경쟁력 향상

5. **수치 추가**: 백업 크기, 평균 소요 시간, 최근 성공률, 실제 restore 시간, 설정한 보관 기간을 표로 만든다.
6. **RPO/RTO 정의**: 데이터 유형별 목표와 근거를 구분한다.
7. **백업 모니터링 미니 구현**: shell textfile collector 또는 Pushgateway 방식으로 마지막 성공 시각·크기·duration을 노출하고 Grafana·Alertmanager와 연결한다.
8. **OpenStack 운영 증거 보강**: 서비스 상태 확인, VM 생성 흐름, Neutron 장애 진단, 로그 위치를 실습 기록으로 추가한다.
9. **오래된 이력서 표현 정리**: “지각하지 않기”보다 장애 대응, 표준화, 자동화, 복구 가능성처럼 직무 성과 중심 목표로 바꾼다.

## 9. 7일 준비 계획

### 1일차: 사실관계 고정

- 각 기술을 `직접 구현`, `공동 구현`, `환경 이해`, `개선 제안` 네 단계로 분류
- S3 mirror 구현 증거와 AWS S3 사본을 이용한 restore 검증 범위를 구분
- 기여를 입증할 커밋·파일·로그 선정

### 2일차: Linux 운영

- CPU, 메모리, 디스크, 네트워크, systemd 장애 진단을 명령어와 함께 구두 연습
- 실제 장애 사례 1개를 STAR 형식으로 정리

### 3일차: OpenStack

- Keystone-Nova-Neutron-Glance-Cinder 흐름 복습
- 인스턴스 생성 실패와 네트워크 장애 진단 순서 실습

### 4일차: Kubernetes·백업

- Velero, etcd, pg_dump, restic의 복구 단위·한계 비교
- Q31~Q45를 90초 이내로 답변

### 5일차: 모니터링·DevOps

- Prometheus, exporter, Alertmanager, Grafana 데이터 흐름 설명
- GitLab CI와 Argo CD의 역할, 배포 실패 진단 연습

### 6일차: 소통·인성

- 고객 장애 대응, 팀 의견 조율, 실수·에스컬레이션 사례를 STAR로 준비
- 60초 자기소개 녹음 후 불필요한 수식어 제거

### 7일차: 실전 모의면접

- 기본 질문 10개, 기술 꼬리질문 10개, 압박 질문 5개
- 답변마다 주장-근거-한계-개선 순서가 있는지 점검

## 10. 답변 원칙

1. **결론부터 말한다.**
2. **내가 한 행동을 동사로 말한다.** “참여했다”보다 “분류했다, 설정했다, 검증했다, 문서화했다”가 좋다.
3. **팀 결과와 개인 기여를 분리한다.**
4. **검증하지 않은 것은 검증했다고 말하지 않는다.**
5. **한계를 말한 뒤 운영 개선안으로 연결한다.**
6. **기술 선택은 도구 이름이 아니라 장애·복구 요구사항으로 설명한다.**
