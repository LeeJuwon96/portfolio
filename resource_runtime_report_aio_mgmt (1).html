0) K8s 접근 경로 보강 (vmv 기반 확인)

요청한 “쿠베 컨피그가 .kube에 있음” 조건을 반영하여 수집 경로를 재정리했습니다.

acer-aio 확인

  1. 접근 사용자: ubuntu@172.16.1.10 (vmv/원격 세션 경유)
  2. ~/.kube 존재 확인: acer-kubeadm.yaml 파일 존재
  3. 주의: 기본 경로 ~/.kube/config 미사용 시 `kubectl`이 localhost(127.0.0.1:8080)로 조회되어 오탐/연결 실패 가능
  4. 권장 점검: 아래 방식으로 컨텍스트를 명시 조회
export KUBECONFIG=$HOME/.kube/acer-kubeadm.yaml
kubectl --kubeconfig "$KUBECONFIG" config get-contexts -o name
kubectl --kubeconfig "$KUBECONFIG" config current-context
kubectl --kubeconfig "$KUBECONFIG" cluster-info

acer-mgmt 확인

  1. 접근 사용자: user1@acer-mgmt

  3. Node: k3d-mgmt-server-0 Ready

8) 분석 및 권고

AIO

  1. 최우선 이슈: VM 엔진(qemu) 3개가 CPU/메모리를 상위 점유. 현재 Load Avg는 CPU 초과 가능성이 높음. VM 수요가 피크인지 확인 필요.
  2. 권고: RabbitMQ 상태 확인 스크립트(307% 점유)와 같은 순간성 프로세스를 제외한 기준으로 재측정하여 장기 평균을 판단.
  3. 권고: docker socket 권한 정책 정비(현재 사용자 비특권 접근 제한). 모니터링/오토메이션 계정에 docker 그룹/권한 명시 필요.
  4. 권고: kubectl 관리 오케스트레이션은 --kubeconfig ~/.kube/acer-kubeadm.yaml으로 강제 지정 후 재측정.
  5. 권고: context가 정상인지 먼저 확인(예: kubectl --kubeconfig ~/.kube/acer-kubeadm.yaml config get-contexts).
  6. 권고: virsh 미설치면 Libvirt 관리가 불가하므로 VM 가시성 확보를 위해 별도 도구(예: OpenStack CLI/API or libvirt 패키지) 통합.

MGMT

  1. 핵심 포인트: Node/파드 자체 상태는 안정적이나 컨테이너 메모리 집적이 큼. GitLab, SonarQube, Elasticsearch, Logstash, Kafka가 메모리 상위권.
  2. 권고: 각 컨테이너의 requests/limits 정책 재정의(특히 logstash, sonarqube, elasticsearch, kafka).
  3. 권고: gitlab/kafka/suricata 계열 JVM 컨테이너는 Heap/Metaspace 정책 점검으로 OOM 및 GC 흔들림 방지.
  4. 권고: tailscale/bridges 다량 네트워크 인터페이스는 실사용 트래픽 상한 및 MTU/오버헤드 정합성 추가 검토.
  5. 권고: 매 5분 간격 스냅샷 + 5일치 시계열 보관으로 추세 기반 임계치 알람 구축.
